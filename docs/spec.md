# 災害時医療・インフラ予測マップアプリ（仮称） 仕様書

## 1. 概要 (Overview)
本アプリケーションは、平時および災害時における地域の医療ニーズとインフラ（避難所・トイレ）の状況を可視化するWebアプリケーションである。
国や自治体が公開するオープンデータ（人口密度メッシュ、医療機関、避難所、公衆トイレ）を掛け合わせ、住所検索から1ステップで「周辺のリスク（人口密度ベースの混雑予測）」と「最適な医療・インフラ資源」を地図上に色分けして表示する。

## 2. ターゲットユーザーとユースケース (User Stories)
*   **一般住民（持病あり / 要配慮者家族）**
    *   災害時に「自分の持病に対応できる最も近い災害拠点病院」がどこか、またその周辺がどれくらい混雑しそうかを平時から把握する。
    *   避難所周辺に「多機能トイレ（車椅子・オストメイト対応など）」があるかを事前に確認する。
*   **地域防災計画の策定者 / 自治体関係者**
    *   人口密度に対して、避難所や医療機関、公衆トイレの配置が不足している「潜在的空白地帯」を机上でシミュレーションする。

## 3. システムアーキテクチャ (Architecture)
Azureのサーバーレスアーキテクチャを採用し、低コストかつ運用の手間のない構成とする。

```
[ Frontend: Azure Static Web Apps (Free) ]
│ Vanilla JS (ESM・ビルド不要) + Azure Maps Web SDK v3
│
│ CORS（許可オリジンは SWA の1つのみ）
▼
[ Backend: Azure Functions (独立リソース / Linux 従量課金) ]
│ Python 3.11 / V2 Programming Model
├─► [ Map ]       Azure Maps Web SDK（地図描画のみ）
├─► [ Geocoding ] 国土地理院 住所検索API（キー不要）
├─► [ 生成AI ]    OpenAI API（切替可: Azure OpenAI / スタブ）
└─► [ Data ]      加工済み JSON（Function App に同梱）
```

### コンポーネント選定理由:
*   **Azure Static Web Apps (SWA):** Freeプランによる運用コストの最小化とHTTPSの自動化。
    フロントはビルド不要の Vanilla JS のため、静的ファイルをそのまま配信している。
*   **Azure Functions は SWA 内蔵ではなく独立リソース:**
    当初は SWA のマネージドAPIを使う設計だったが、**マネージドAPIの Python は 3.8 までしか対応しておらず**、
    本プロジェクト（3.10+）では動作しない。3.9/3.10 での不具合報告もあり、
    公開後に動かないリスクを避けて独立した Function App (Python 3.11) に変更した。
    SWA の「リンクバックエンド」機能なら同一オリジンにできるが Standard プラン（有料）が必要なため、
    Free プランのまま CORS で連携している。
*   **Azure Maps:** 大容量のメッシュデータ・ピンを高パフォーマンスで描画するため（地図描画のみに使用）。
*   **ジオコーディングは国土地理院API:** Azure Maps Search は日本の住所をほぼ解決できなかった（実測）。
    県庁の住所すら 0 件または 9〜15km ずれるため、防災用途では採用できないと判断した。
    国土地理院は同等の住所を 0.01〜1.4km で解決できる。実装は差し替え可能な形で両方保持している。
*   **生成AI は OpenAI API:** Azure OpenAI はデプロイ単位のクォータ割当が必要で運用が煩雑なため。
    `LLM_BACKEND` の切替で Azure OpenAI にも戻せる。
*   **データは Blob Storage ではなく Function App に同梱:**
    加工済み JSON は全体で約 8MB と小さく、更新頻度も年単位（オープンデータの改訂時のみ）。
    Blob を挟むとリクエストごとに外部I/Oとレイテンシが増えるため、同梱して即座に読む方式にした。
    `DataStore` を抽象化してあるため、データ量が増えた場合は `DATA_STORE_BACKEND` の切替で Blob へ移行できる。
    なお 125m メッシュ（12MB）はサイズと転送量の割に用途が限られるため、公開版では配信していない。

### 3.1. 公開時のセキュリティ・コスト対策
認証を設けず一般公開するため、従量課金リソースの濫用対策を実装している。

*   **`/api/chat` のレート制限** … IP単位 10回/時間（移動窓）＋ 全体 200回/日。
    後者はIPを変えられても効く上限で、課金に対する最後の砦となる。
    カウンタはプロセス内メモリのため、Function App のスケール上限を **1インスタンスに固定**して整合性を担保している。
    本番規模では Redis 等の共有カウンタへ差し替える前提で `RateLimiter` を抽象化している。
*   **Azure Maps キーの参照元制限** … Maps アカウントの CORS ルールで、公開URLからの利用のみ許可。
    キーはブラウザに渡る性質上抜き取りが避けられないため、抜かれても他サイトで使えない状態にしている。
*   **Function App の CORS** … 許可オリジンを SWA の1つに限定（ワイルドカードは削除済み）。
*   **予算アラート** … 月額の閾値超過時と超過予測時にメール通知。
    ただし通知は停止を伴わないため、**LLM の実質的な上限は OpenAI 側の利用上限設定**で担保する。

## 4. データ設計 & オープンデータ要件 (Data Ingestion)
各種オープンデータを事前にローカル環境で加工（クレンジング・結合）し、以下の構造のJSONファイルとして
`data/processed/` に出力する。生成物は Function App に同梱して配信する（3章の選定理由を参照）。
加工スクリプトは `scripts/clean_*.py`、読み込みは `DataStore` 抽象を経由するため、
`DATA_STORE_BACKEND` の切替で Blob Storage へ移行できる。

### 4.1. 人口密度データ（1km / 500m / 250m / 125mメッシュ）
*   **データ元:** 総務省国勢調査（e-Stat 統計GIS）
*   **実装状況:** 4解像度に対応。`scripts/clean_mesh.py` で生成し、`/api/mesh?resolution=` で切替。
    対象エリアは**神奈川県全域**。
    | 解像度 | KEY_CODE桁数 | セル数 | 公開版で配信 |
    |---|---|---|---|
    | 1km | 8桁 | 1,852 | ○（既定） |
    | 500m | 9桁 | 6,250 | ○ |
    | 250m | 10桁 | 20,896 | ○ |
    | 125m | 11桁 | 68,980 | ×（12MB のため除外） |
    選択肢は `available_mesh_resolutions()` が実ファイルの有無から組み立てるため、
    125m を配信対象から外すと UI からも自動的に消える。
*   **解像度は元データで決まる:** KEY_CODE の桁数が解像度を表すため、
    1km データから 500m を計算で作ることはできない。
*   **スキーマ案:**
    ```json
    {
      "meshId": "53393581", 
      "coordinates": [[lng1, lat1], [lng2, lat2], ...], 
      "populationDensity": 8500 
    }
    ```

### 4.2. 医療機関データ（災害拠点病院・一般病院）
*   **データ元:** 厚生労働省「医療情報ネット」（施設情報 01-1 / 診療科目 01-2 の2ファイルを ID で結合）
*   **実装状況:** 神奈川県 310 施設。`scripts/clean_hospitals.py` で生成。
*   **未充足:** 元データに災害拠点病院の区分が無い（`機関区分` は全件「1」＝病院）ため
    `isDisasterBase` は false 固定。`topDiseases` も元データに無いため空配列。
    実現には自治体公開の災害拠点病院一覧を別途取り込む必要がある。
*   **スキーマ案:**
    ```json
    {
      "id": "hosp_001",
      "name": "〇〇総合病院",
      "isDisasterBase": true, 
      "location": { "lat": 35.514, "lng": 139.718 },
      "capabilities": ["循環器系", "糖尿病", "人工透析"], 
      "topDiseases": ["心不全", "誤嚥性肺炎", "脳梗塞"] 
    }
    ```

### 4.3. 避難所・福祉避難所データ
*   **データ元:** 国土地理院「指定緊急避難場所」「指定避難所」（神奈川県）
*   **実装状況:** 4,265 施設（緊急避難場所 3,332 / 指定避難所 1,486 / 兼用 553 / 福祉避難所 151）。
    `scripts/clean_shelters.py` で生成。
*   **重要:** 避難所は災害時の役割がまったく異なる2種類に分かれ、混同すると命に関わる。
    *   **指定緊急避難場所**: 切迫した危険から「命を守る」場所。災害種別ごとに指定される。
    *   **指定避難所**: 自宅に戻れない人が一定期間「生活する」場所。災害種別の指定は無い。
    同一施設が両方を兼ねることがあるため、2つの真偽値で表現する。
*   **スキーマ:**
    ```json
    {
      "id": "evac_E1413000000001",
      "name": "〇〇小学校体育館",
      "address": "神奈川県川崎市…",
      "location": { "lat": 35.512, "lng": 139.715 },
      "isEmergencySite": true,
      "isEvacuationCenter": true,
      "isWelfareShelter": false,
      "disasterTypes": ["flood", "earthquake"],
      "targetOccupants": ""
    }
    ```
*   **災害種別（災害対策基本法の8種別）:** flood(洪水) / landslide(崖崩れ等) / stormSurge(高潮) /
    earthquake(地震) / tsunami(津波) / fire(大規模な火事) / inlandFlood(内水氾濫) / volcano(火山現象)
*   **元データの罠:** 配布ディレクトリ名と中身が入れ違っているため、
    ファイル名ではなく列構成で種別を判定している。

### 4.4. 公衆トイレデータ
*   **データ元:** OpenStreetMap（Overpass API）
*   **実装状況:** 神奈川県 1,953 件（うち車椅子対応 322 件）。`scripts/fetch_toilets.py` で取得。
*   **スキーマ案:**
    ```json
    {
      "id": "toilet_001",
      "name": "〇〇公園公衆便所",
      "location": { "lat": 35.513, "lng": 139.716 },
      "attributes": {
        "accessible": true, 
        "ostomate": true,   
        "open24h": true     
      }
    }
    ```

### 4.5. 地価データ
*   **データ元:** 国土交通省 国土数値情報「地価公示」(L01-21_14 / 2021年 / 神奈川県)
*   **実装状況:** **神奈川県全域** 1,787 地点・1,178 町名。`scripts/clean_landprice.py` で生成。
    価格帯は 1,170 〜 15,600,000 円/㎡（中央値 190,000 円/㎡）。
*   **町名別の集計はビルド時に行う:** 「地価が安く避難所が近いエリア」をAIに答えさせるには
    町名ごとの集計が要るが、リクエスト時に計算すると 1,787地点 × 4,265避難所 の距離計算で
    **51秒**かかった。`landprice_areas.json` として事前生成し、0.18秒に短縮している。
*   **色分けは分位点で区切る:** 価格帯が4桁の幅を持つため、最小〜最大の線形分割では
    99.8% が最下位の色帯に入り、ほぼ単色の地図になった。
    分位点（5/30/55/80/95%）で区切り、各帯が 24.7 / 25.2 / 25.1 / 15.0 / 5.0% に分散している。
*   **スキーマ:**
    ```json
    {
      "id": "land_14131_0001",
      "address": "神奈川県川崎市川崎区中瀬2-17-12",
      "town": "中瀬",
      "location": { "lat": 35.52128, "lng": 139.67347 },
      "pricePerSqm": 272000,
      "uses": ["住宅"],
      "year": "2021"
    }
    ```
*   **注意:** 地価公示は「地点」の価格であり、町丁目全体の相場ではない。
    同じ町でも地点により差があるため、エリアの傾向を示す目安として扱う。
    また地価の安い地点は工業地帯や浸水想定区域であることがあり、
    安全性を地価だけで判断してはならない。

## 5. 機能要件 (Functional Requirements)
*   **5.1. 住所検索機能:** ユーザーが任意の住所を入力。Backend（Python）で **国土地理院 住所検索API** を叩き、該当箇所の中心緯度経度を取得して地図を移動。
    当初は Azure Maps Search を想定していたが、日本の住所を解決できなかったため差し替えた（3章参照）。
*   **5.2. 地図表示・オーバーレイ機能:** 周辺メッシュデータを読み込み、人口密度に応じて地図のエリア自体を色分け表示（高：赤色 〜 低：緑色）。中心点から一定半径内のインフラピンを表示。
*   **5.3. 疾患・条件フィルタリング機能:** 画面上で「持病（例：糖尿病、循環器など）」を選択した際、対応する病院ピンを強調表示。トイレも「多機能トイレのみ」のフィルタリングを可能とする。
*   **5.4. 詳細情報ポップアップ（モーダル）:** 地図上のピンをクリックした際、詳細情報（病院の強み、避難所の区分、トイレの設備など）をモーダル表示。
*   **5.5. 基準地点の指定:** 距離計算とAIへの質問の起点。以下の3通りで指定でき、いつでも切替可能。
    *   起動時ダイアログで**住所入力**（国土地理院 住所検索API）
    *   **ブラウザの現在地**（`navigator.geolocation`）
    *   **既定地点**（神奈川県庁）。住所・現在地とも指定しない場合のフォールバック。
    対応エリアは神奈川県のみ。県外の住所が入力された場合は 422 で対象外を伝える。
*   **5.6. 生成AI防災アシスタント:** チャット欄から自然言語で質問できる。
    サーバが基準地点の周辺施設（避難所・医療機関・トイレ）と地価エリア集計を
    プロンプトに組み立てて LLM へ中継する。回答文に登場した施設は地図上に旗で強調する。
    *   数値の集計（距離・地価の平均・施設数）は**サーバ側で行い**、LLM には計算させない。
    *   LLM に施設IDを列挙させると存在しないIDを生成する恐れがあるため、
        「サーバが渡した候補の名前が回答文に現れたか」で強調対象を判定する。
*   **5.7. 地価と防災施設の掛け合わせ:** 地価の上限をスライダーで指定し、
    条件を満たすエリア・地点のみを強調表示する。AIは町名ごとの
    「平均地価」と「徒歩圏（半径0.8km）の避難所・医療機関数」から
    コストパフォーマンスの良いエリアを提案する。

## 6. 非機能要件 (Non-Functional Requirements)
*   **パフォーマンス:** 住所入力から地図描画、ピン配置完了まで 2秒以内。
*   **モバイルファースト:** スマホ利用を想定し、レスポンシブデザイン（Tailwind CSS等を推奨）を徹底。
*   **コスト最適化:** 待機コストを 0 円に近づけるサーバーレス構成（従量課金）とする。

## 7. 品質管理・開発プロセス仕様 (Quality Assurance & DevOps)
本プロジェクトでは、コードの品質担保とリファクタリングスキルの向上のため、Claude Codeを活用した自動コードレビューおよびテスト駆動開発（TDD）の要素を組み込む。

### 7.1. AIコードレビュー基準 (Code Review Guidelines)
Claude Codeにコードの解説や修正を求める際、以下の観点でのレビューを必須とする。

*   **Azureベストプラクティス:**
    *   Azure Functions（Python）のコールドスタートを意識した、メモリ消費の少ない軽量な外部ライブラリ選定になっているか。
    *   Azureの資格情報（APIキーや接続文字列）がハードコードされず、環境変数（`os.environ`）から安全に読み込まれているか。
*   **パフォーマンスとメモリ管理:**
    *   地図上に大量のピン（病院やトイレ）をプロットする際、DOM要素を直接増やさず、Azure Mapsのデータレイヤー（SymbolLayer）を正しく使っているか。
    *   フロントエンドでの不要な再レンダリングが発生していないか。
*   **例外処理（エラーハンドリング）:**
    *   外部API呼び出し（`requests` 等）やデータ欠損時に画面が真っ白にせず、適切な例外処理とユーザーへのエラーメッセージ（「データが取得できませんでした」等）を表示すること。
*   **コード提示ルール:**
    *   修正コードを提示する際は、一部の抜粋ではなく、必ず過去のソースを鑑みた**ファイルのフルコード（全コード）**を提供すること。

### 7.2. テスト戦略 (Testing Strategy)
本アプリの信頼性を担保するため、以下の2つのレイヤーでテストを自動化する。

*   **バックエンド（Azure Functions / スクリプト）:**
    *   `pytest` による単体テスト。住所検索（ジオコーディング）のレスポンス処理、データクレンジングロジック、フィルタロジックの検証。
*   **フロントエンド:**
    *   `Vitest` + `Testing Library` 等によるUIコンポーネントテスト。チェックボックス等の操作でピンの表示・非表示が正しく動作するか等を検証。

### 7.3. Claude Codeへの開発指示フロー (Developer Workflow)
開発時は、常に以下の「3ステップ」のサイクルを意識してClaude Codeと対話する。

1.  **[Red] テストの作成を先に依頼する:**
    *   例: 「今から〇〇の機能を作りたい。まずはその機能が満たすべきテストコードを先に書いて」
2.  **[Green] テストを通すコードを書かせる:**
    *   例: 「このテストがすべてパスするように、本体のコードを実装して」
3.  **[Refactor & Review] レビューとリファクタリングを求める:**
    *   例: 「テストは通った。7.1の基準に照らし合わせて、このコードに改善の余地（リファクタリングやセキュリティの穴）がないかレビューして、修正版を**ファイルのフルコード**で提示して」
## 8. デプロイ構成 (Deployment)

### 8.1. Azure リソース
すべて単一のリソースグループ `rg-bousai-map` に配置する。

| 種別 | 用途 | プラン |
|---|---|---|
| Static Web Apps | フロントエンドの配信 | Free |
| Functions (Linux) | API（Python 3.11） | 従量課金・**スケール上限 1** |
| Storage Account | Functions の実行基盤 | Standard_LRS |
| Maps アカウント | 地図描画 | G2 |

スケール上限を 1 に固定しているのは、レート制限のカウンタがプロセス内メモリのため（3.1参照）。
性能上の制約ではなく、課金対策の整合性を取るための設定である。

### 8.2. 設定値
資格情報を含む設定はすべて Function App のアプリケーション設定（環境変数）から読み込む。
リポジトリには一切含めない。

| 変数 | 内容 |
|---|---|
| `AZURE_MAPS_SUBSCRIPTION_KEY` | 地図描画用。`/api/config` 経由でフロントへ渡す |
| `OPENAI_API_KEY` / `OPENAI_MODEL` | 生成AI |
| `LLM_BACKEND` / `GEOCODER_BACKEND` / `DATA_STORE_BACKEND` | 各抽象の実装切替 |
| `CHAT_RATE_LIMIT_PER_IP` / `_WINDOW_SECONDS` / `_DAILY_GLOBAL` | レート制限（0 で無制限） |

### 8.3. デプロイ手順

```bash
# API
func azure functionapp publish <FUNCAPP_NAME> --python --build remote

# フロントエンド
./scripts/deploy_frontend.sh
```

`frontend/index.html` の `<meta name="api-base">` は**リポジトリ上では空**にしておく。
埋めるとローカル開発時にも本番APIを参照してしまうため、`deploy_frontend.sh` が
配信用の一時コピーにのみ Function App のURLを差し込む方式とする。
差し込みに失敗した場合はデプロイを中断する（そのまま配信すると全APIが404になり、
画面上は「データが取得できませんでした」としか出ず原因究明が困難なため）。

配信対象から `frontend/tests/` は除外する。デプロイ対象からは `data/raw/`（83MB）と
125mメッシュ（12MB）を `.funcignore` で除外し、約8MBに収めている。
